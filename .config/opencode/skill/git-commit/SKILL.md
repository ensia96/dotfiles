# Git Commit

## 형식

```
{type}: {message}
```

## 작성 원칙

- **한글 필수**: type만 영문, message는 한글
- **자기 완결적**: commit title만으로 변경 내용 파악 가능
- **자연스러운 명확성**: message가 모호하면 대상 명시를 고려하되, 억지로 대상을 붙이기보다 커밋 단위 분리를 먼저 검토

## Types

| Type | 용도 |
|------|------|
| `init` | 새로운 것 추가 |
| `feat` | 기존 것 변경 |
| `fix` | 버그 수정 |
| `docs` | 문서/주석 |
| `chore` | 동작에 영향 없는 변경 |

## 커밋 단위 결정

**기본 원칙**: 하나의 커밋 = 하나의 독립적 변경

| 질문 | "예"면 분리 |
|------|------------|
| 이 변경만 따로 되돌릴 필요가 있을 수 있는가? | 별도 커밋 |
| 변경들이 서로 다른 목적인가? | 별도 커밋 |
| 설정 변경과 그로 인한 결과인가? | 별도 커밋 |

| 질문 | "예"면 합침 |
|------|------------|
| 하나가 없으면 나머지가 의미 없는가? | 함께 커밋 |

**모호한 상황 예시**:
- `.gitignore` 변경 → 추적 대상 증가: **분리** (설정 변경 vs 파일 추가)
- 리팩토링 → 여러 파일 동시 수정: **합침** (하나의 논리적 변경)

## Message 작성

| 상황 | 접근 |
|------|------|
| 커밋 단위가 명확함 | message만으로 충분 |
| 커밋 단위가 넓음 | 대상 명시 고려, 또는 단위 분리 검토 |

```
# 단위가 명확한 경우
init: 로그인 페이지 추가
fix: 무한 루프 해결

# 대상 명시가 자연스러운 경우
feat: git-commit 스킬에 커밋 단위 기준 추가
feat: .gitignore에 skill 디렉토리 whitelist 추가
```

## 저장소별 스타일

**커밋 전**: 기존 커밋 이력 확인

```bash
git log --oneline -20
```

| 구분 | 기준 |
|------|------|
| **양식** | 현재 스킬 (`{type}: {message}`) |
| **내용** | 기존 이력의 표현 방식, 어휘 참조 |

## 커밋 전 확인

- [ ] 한글로 작성했는가? (type 제외)
- [ ] 제3자가 이 메시지만 보고 변경 내용을 이해할 수 있는가?
- [ ] 커밋 단위가 적절한가? (너무 넓으면 분리 검토)
- [ ] 민감 정보가 포함되지 않았는가?

## 금지

- 영문 message
- 모호한 message (대상 명시 또는 단위 분리로 해결)
